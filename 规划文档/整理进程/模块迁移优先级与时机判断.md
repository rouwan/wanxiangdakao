# 模块迁移优先级与时机判断

> 日期：2026-06-14  
> 背景：四个子模块中仅有前台前端（mpvue）无法运行，其余三个模块当前可用。本文档记录分级迁移策略及决策依据。

---

## 当前状态

| 模块 | 可运行 | 框架状态 | 迁移优先级 |
|------|--------|----------|-----------|
| 前台前端（小程序） | ❌ | mpvue 已死 | 🔴 必须立即迁移 |
| 后台前端（管理页） | ✅ | Vue 2 可用，Element UI 可用 | 🟡 条件触发迁移 |
| 后台后端（管理 API） | ✅ | Spring Boot 2.0 EOL 但可用 | 🟡 条件触发迁移 |
| 前台后端（游戏 API） | ✅ | Spring Boot 2.0.3 EOL 但可用 | 🟡 条件触发迁移 |

---

## 为什么现在不搬其他三个

不是因为"它们没问题"，而是因为**现在还能跑，而维护成本尚未超过迁移成本**。

### 后台前端（Vue 2）—— 靠补丁活着

当前的运行状态依赖于一系列手工适配：

- `node-sass` → Dart Sass（前者在 Node 14+ 无法安装）
- `build/utils.js` 注入 `implementation: require('sass')`
- 全局安装 Node 14 便携版（Node 24 不兼容老 Webpack 3）
- `npm install --legacy-peer-deps`（绕过新版 npm 的 peer dependency 检查）

**每一条都是隐性债务**：下次系统重装、Node 升级、或需要重新 `npm install` 时，这些补丁可能失效且排查困难。

### 后台后端 + 前台后端（Spring Boot 2.0）—— 依赖供应链在退化

- Spring Boot 2.0 已于 2019 年 EOL，安全漏洞持续增加
- `jave:1.0.2` 等依赖已从 Maven Central 下架（本次点亮时已删除此依赖）
- Spring Boot 2.0 → 3.x 不是升级路径，是重写路径（Java 8 → 21，javax → jakarta）
- JDK 8 更新支持到 2026 年 11 月终止
- 两个后端共享同一数据库，实际是连体婴儿——要动就得一起动

### 核心逻辑：维护成本曲线

```
旧依赖的可用性随时间递减：
  今天：Spring Boot 2.0 核心依赖全在 Maven Central
  半年后：可能有包被标记 deprecated
  一年后：可能有关键包被移除
  两年后：JDK 8 停止安全更新，Spring Boot 2.0 生态基本不可修复

维护成本 = 基础工作量 + 随时间增长的"腐烂成本"
腐烂成本来自：依赖消失、Node/Java 版本不兼容、安全漏洞累积
```

**当前"腐烂成本"还低，预计 6-12 个月内达到临界点。**

## 触发迁移的两个信号

满足**任意一个**就应该执行迁移：

### 信号一：需要加新功能

如果要在旧模块上开发新功能，但旧栈的研发效率显著低于新栈：

| 示例 | 旧栈成本 | 新栈成本 |
|------|---------|---------|
| 加一个 REST 接口 | 手写 JDBC → 拼 JSON String → 改 5 个文件 | Spring Boot 3 + MyBatis-Plus 自动映射 |
| 管理后台加一个列表页 | Vue 2 + 老 Element UI，组件 API 过时 | Vue 3 + Element Plus，生态活跃 |
| 加一个 WebSocket 消息类型 | 无类型安全，纯手写协议解析 | 新栈可定义 message schema |

**原则**：如果旧栈开发成本 > 新栈迁移成本，直接做迁移而不是在旧栈上加功能。

### 信号二：旧栈某个关键依赖坏了

- npm registry 标记某个 Vue 2 关键包为 deprecated 且不可安装
- Maven Central 移除 Spring Boot 2.0 的核心依赖
- Node 14 便携版下载链接失效且无法获取
- 操作系统/安全策略强制升级，旧环境无法运行

**此时修补旧栈的成本 ≈ 在新栈上重写的成本**，直接做迁移。

### 都不满足时

**不动它。** 旧栈能跑就继续跑，精力全部放在小程序重写上。

## 推荐的迁移节奏

### 第一阶段（现在）

```
万象大考v3/
└── dakao/              # Uni-app 小程序，通过 API 调旧后端
```

小程序是用户唯一能感知的东西，先把它重写完上线。

### 第二阶段（条件触发）

```
万象大考v3/
├── dakao/              # Uni-app 小程序
└── admin-web/          # Vue 3 + Element Plus 管理前端
```

管理前端等第一个触发信号。迁移时可以直接参考旧管理前端的页面结构和 API 调用。

### 第三阶段（条件触发）

```
万象大考v3/
├── dakao/              # Uni-app 小程序
├── admin-web/          # Vue 3 + Element Plus 管理前端
└── dakao-api/          # Spring Boot 3 单体（合并两个旧后端）
```

后端迁移放最后，因为：
1. 两个后端共享数据库，必须一起动
2. API 协议不变时前端不受影响
3. 数据迁移（60+ 张表）需要充分验证

## 总结

| 问题 | 答案 |
|------|------|
| 为什么现在只搬小程序？ | mpvue 是唯一物理死亡、无法抢救的模块 |
| 其他三个为什么不现在搬？ | 能跑。当前维护成本 < 迁移成本 |
| 什么时候搬？ | 要加新功能但旧栈效率低，或旧依赖坏了修不好 |
| 怎么判断"修不好"？ | 修补旧栈的时间 > 在新栈上重写的时间 |
| 最晚什么时候必须搬？ | JDK 8 安全更新终止（2026 年 11 月）之前 |
| 先搬哪个？ | 管理前端（独立、改动量小） > 后端（两个连体、数据迁移） |
