## 一、项目概述

**万象大考**是一个微信小程序答题对战游戏平台。用户通过微信小程序进行：

- 🎮 **1v1 随机匹配对战**（排位赛，段位制，9 段位）
- 🏠 **房间对战**（4 人混战 / 3v3 组队）
- 🤖 **AI 机器人补位**（无人匹配时自动补位，多难度等级）
- 📊 **多维度排行榜**（段位、战力、连胜、单局最高分、好友榜）
- ✍️ **用户出题工厂**（众包题库，审核上架）
- 🎁 **签到大礼包**（每日签到、撸猫、幸运号码、转盘抽奖）
- 📚 **知识宝典**（6 大学科升阶系统，答对加分）
- 💪 **体力/精力系统**（上限 30，每 120 秒恢复 1 点）

平台同时管理多个小程序模块（统称为"small"）：

| 模块 | URL 前缀 | 说明 |
|------|----------|------|
| wxdk | `/smallVientianeTest` | **万象大考**（主体） |
| ktcc | `/smallLookGuess` | 看图猜词 |
| csjk | `/smallLongHealth` | 视频/健康 |
| PYGX | `/friendrelationship` | 朋友关系 |
| SQCS | `/smallMagictest` | 神奇测试 |

## 二、整体架构

```
微信小程序（mpvue）
    │  HTTP + WebSocket
    ▼
┌─────────────────────────────────────┐
│  用户端 API (Spring Boot 2.0.3)     │  端口 8008
│  vientianetestappendapi             │
│  - 登录/授权/用户信息               │
│  - 排位对战 WebSocket (wsRank)      │
│  - 房间对战 WebSocket (wsRoom)      │
│  - 题库/宝典/签到/抽奖 REST API     │
└──────────────┬──────────────────────┘
               │ JDBC
               ▼
┌─────────────────────────────────────┐
│  MySQL 5.7 (101.132.104.79:3306)    │
│  数据库: smalldo / wxdkgamedo       │
│  + Redis (localhost:6379)           │
└──────────────┬──────────────────────┘
               │ JDBC (同一数据库)
               ▲
┌──────────────┴──────────────────────┐
│  管理端 API (Spring Boot 2.0.0)     │  端口 8005
│  renren-fast                        │  context-path: /renren-fast
│  - Shiro + JWT 认证                 │
│  - RBAC 权限（用户/角色/菜单）      │
│  - 内容管理 CRUD                    │
│  - Quartz 定时任务                  │
│  - Druid 连接池监控                 │
│  - Swagger API 文档                 │
└──────────────┬──────────────────────┘
               │ HTTP REST
               ▼
┌─────────────────────────────────────┐
│  管理前端 (Vue 2.5 + Element UI)    │
│  renren-fast-vue-master             │
│  - 动态菜单/权限控制                │
│  - 题库/机器人/商品/抽奖管理        │
└─────────────────────────────────────┘
```

### 关键架构特点

1. **两个 Java 后端共享同一数据库**。`renren-fast` 做管理 CRUD，`vientianetestappendapi` 做游戏逻辑 + WebSocket。两者没有直接的 RPC 调用，通过数据库和 Redis 交换状态。
2. **管理后端是 renren-fast 脚手架二次开发**。sys 模块用 MyBatis-Plus（标准 RBAC），业务模块（`small/*`）用原生 `NamedParameterJdbcTemplate` 拼 SQL。
3. **游戏后端自研**。没有任何 ORM 框架，全部手写 JDBC + `BasicResultObject` + `StringBuilder` 拼 JSON 返回。
4. **WebSocket 是游戏核心**。两个独立 WebSocket 端点（`/wsRank` 排位、`/wsRoom` 房间），分别有独立的 Handler + Service + 定时任务驱动匹配和机器人。

## 四、核心数据流

### 4.1 登录流程

```
小程序 wx.login() 获取 code
    → GET /vientianetest/mobile/login?code=xxx
    → 后端调用微信 code2session 获取 openId
    → 新用户: 初始化 user + ability + combat + paragraph + gameStatis
    → 老用户: 返回 userId + 加密 code
    → 小程序存 SessionId（Cookie），后续请求带 JSESSIONID
```

### 4.2 排位对战流程（WebSocketRank）

```
1. 用户连接 wsRank?code=xxx → 解密 userId，创建 User 对象加入 userMap
2. 发送 type=1 → 进入匹配队列 matchingUsers
3. TaskTimer.runMatch() 每秒执行匹配:
   - 等待 5 秒收集玩家
   - 按段位范围匹配（±1-2 段）
   - 避免最近对手，避免 3 局内重复
   - 无人匹配 / 第 5 局 / 连输 3 局 → 补机器人
4. 匹配成功 → 发送 type=1（双方信息）到双方
5. 双方发送 type=3 确认开始 → 创建 Game，推送第一题
6. 每题 10 秒，type=4 提交答案:
   - 正确: 100 基础分 + (剩余秒数/10) 速度分
   - 最后 1 题: 基础分 * 30 暴击
   - 知识宝典加成: bookScore 额外加分
7. type=6 请求下一题 → 发下一题或结束
8. 结束: type=7 结算 → 更新段位/战力/连胜/金币
```

### 4.3 房间对战流程（WebSocketRoom）

```
1. 用户连接 wsRoom → 进入房间大厅
2. type=13 创建房间（4 位随机 roomId）
3. 其他用户 type=16 加入房间（通过分享链接或房间号）
4. type=10 准备/取消准备
5. type=14 快速开始：
   - 人满直接开
   - 不满进入匹配队列 → 合并房间 → 补机器人
6. 游戏流程同排位（5 题，每题 10 秒）
7. 3v3 模式: modeType=2，分左右两队，团队总分决胜
8. 结束后玩家回到原房间，机器人回池
```

## 五、定时任务体系

### 5.1 游戏后端 TaskTimer（核心驱动，1 秒精度）

| 任务 | 间隔 | 功能 |
|------|------|------|
| 排位_匹配玩家 | 1s | `webSocketRankService.runMatch()` — 遍历匹配队列，找人/机器人配对 |
| 排位_自动游戏 | 1s | `webSocketRankService.runRobot()` — 机器人延时作答，模拟人类行为 |
| 排位_判断超时 | 1s | `webSocketRankService.runTimeOut()` — 答题超时 10s 强制跳过 |
| 房间_匹配玩家 | 1s | `webSocketRoomService.runMatch()` — 房间模式匹配 |
| 房间_机器人自动答题 | 1s | `webSocketRoomService.runRobot()` — 房间机器人作答 |
| 房间_判断超时 | 1s | `webSocketRoomService.runTimeOut()` — 房间超时判定 |
| 修改题目正确率 | 3s | `webSocketRankService.runSubjectRatio()` — 批量更新题目正确率统计 |
| 排位_心跳超时 | 10s | 120s 无心跳踢出 |
| 房间_心跳超时 | 10s | 60s 无心跳踢出 |

### 5.2 游戏后端 ScheduledTask（Cron 表达式）

| 时间 | 功能 |
|------|------|
| 每月 1 日 00:00 | 清空段位排行榜 ZSet，重置段位 |
| 每日 02:00 | 清空战力榜 + 连胜榜，周一发邮件，清空房间赢榜 |
| 每日 17:00 | 发微信模板消息提醒"撸猫" |

### 5.3 管理后端 Quartz

Renren-fast 内置 Quartz，通过 `/sys/schedule/**` 接口管理定时任务的 CRUD / 运行 / 暂停 / 恢复。

## 七、技术债务汇总

| 类别 | 问题 | 影响 |
|------|------|------|
| **代码质量** | 游戏后端全部手写 JDBC + StringBuilder 拼 JSON | 可维护性极差，字段级修改需要全链路追踪 |
| **数据访问** | 管理后端有两个数据访问模式（MyBatis-Plus + 原生 JDBC）混用 | 新人无法理解什么时候用哪种 |
| **事务管理** | 无声明式事务，未见 `@Transactional` | 数据一致性靠运气 |
| **API 设计** | 返回 JSON 字符串而非对象，Controller 返回类型是 `String` | 无类型安全，Swagger 无法正确生成文档 |
| **前端** | mpvue 框架已于 2019 年停止维护 | 无法在新版微信开发者工具中编译 |
| **管理前端** | Vue 2 + Webpack 3，node-sass 需 Python 2.7 | 现代 Node 环境无法构建 |
| **测试** | 仅 3 个后端测试文件，前端 1 个 HelloWorld 单元测试 | 核心游戏逻辑零覆盖 |
| **日志** | `wxdk_event_log` 表存事件，但无结构化日志框架 | 排查问题只能翻数据库 |
| **配置** | 数据库连接、密钥分散在多个 yml 文件中 | 改一个配置需要改多处 |

## 八、数据库概况

- **数据库名**：`wxdkgamedo`（SQL Dump 中） / `smalldo`（配置文件引用中）——名字不一致需注意
- **字符集**：`utf8mb4_unicode_ci`
- **总表数**：60+ 张（`wxdk_` 前缀约 50 张 + Quartz 11 张 + 系统表 10 张 + ktcc 表 + csjk 表 + 其他）
- **核心表分组**：

| 分组 | 表前缀 | 数量 | 说明 |
|------|--------|------|------|
| 题目 | `wxdk_subject*` | ~10 张 | 题目、选项、分类、正确率统计 |
| 用户 | `wxdk_user*` | ~25 张 | 用户基础信息、能力、战斗力、段位、签到、任务 |
| 游戏 | `wxdk_game*` | ~4 张 | 排位赛记录、房间赛记录 |
| 排行 | `wxdk_gamerank`, `wxdk_rank*` | ~2 张 | 排行榜、赛季时间 |
| 机器人 | `wxdk_robot*` | ~5 张 | 机器人定义、段位、日报 |
| 运营 | `wxdk_daysign`, `wxdk_lucknumber`, `wxdk_prizeratio`, `wxdk_goods`, `wxdk_paragraph` 等 | ~10 张 | 签到、抽奖、商品、段位配置 |
| 配置 | `wxdk_dataconfig`, `wxdk_cleardatatime` | ~2 张 | 系统配置键值对 |
| 其他 | `wxdk_book`, `wxdk_email`, `wxdk_feedback*`, `wxdk_messageboard` | ~5 张 | 宝典、邮件、反馈、留言板 |

- **软删除模式**：大部分表用 `flag` 字段（1=正常, 2=修改, 3=删除），少数表做物理删除
