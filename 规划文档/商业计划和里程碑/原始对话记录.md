# 万象大考 V4 — 原始对话记录

> 记录时间：2026-06-18
> 来源：与 AI 助手的项目讨论会话

---

## 背景

用户持有一个 2018-2019 年的微信小程序“万象大考”（答题对战平台），技术栈为 mpvue + Spring Boot 2.0，已下线。用户希望重启项目，迁移到小游戏平台。

用户已有两份审计文档：《旧代码审计》和《现有功能剖析》，对旧项目的架构、代码、安全问题有完整的认识。

---

## 一、技术架构评估（会话开头）

**用户问**：这个项目很多架构是落后的对吧？

**AI 回答要点**：

- 是的，整体技术栈停留在 2018-2019年
- 严重落后：mpvue已死、Webpack 3、Spring Boot 2.0 EOL、存在 Log4Shell 漏洞
- 明显过时：Vue 2 EOL、fastjson不再推荐、MyBatis-Plus 2.1、renren-fast停维
- 架构层面：两个后端直连同一数据库、全手写 JDBC+StringBuilder拼 JSON、Controller返回 String而非对象、无 CI/CD/无 Git/无容器化、安全配置硬编码
- 但在 2018-2019 年这些选型是合理的，问题是 7 年没维护演进

---

## 二、前端新架构讨论：uni-app 还是 Cocos？

**用户问**：前端新架构是否应该用 uni-app？还是该上游戏引擎如 Cocos？

**AI 回答要点**：

- 对这个项目，uni-app完全够用，Cocos 属于 over-engineering
- 页面构成分析：80% 是表单/列表/展示，只有对战需要“游戏感”
- 对战中的副本画倒计时、动画、WebSocket，uni-app 都能做
- Cocos 什么时候才该上：需要3D、物理碰撞、骨骼动画、拖拽拼图等复杂交互
- 硬上 Cocos 的代价：表单页在 Canvas 里写是灾难、包体积增大、学习成本高、迭代速度慢
- 建议：uni-app + 局部 Canvas 增强

---

## 三、小程序 → 小游戏迁移可行性

**用户问**：我不改动小程序前端，可以直接上小游戏吗？

**AI 回答要点**：

- 不能。小程序和小游戏是微信两个完全不同的运行时
- 小程序用 WebView（WXML+WXSS），小游戏用纯 Canvas（无 DOM）
- mpvue 编译产物是 WXML 模板，小游戏没有 WXML 运行时
- 迁移 = 前端 100% 重写
- 但后端接口逻辑可以复用

**小程序 + 小游戏混合方案讨论**：

- 小程序管理端/展示端 + 小游戏对战端，通过 wx.navigateToMiniProgram() 跳转
- 但代价是两个项目维护、跳转延迟、状态同步复杂
- 对这个项目不建议，纯小程序 Canvas 增强就够

---

## 四、为什么迁移小游戏（用户的三个理由）

**用户提出三个迁移小游戏的核心原因**：

1. 小游戏的激励广告单价高（eCPM ¥30-120 vs 小程序 ¥8-30）
2. 答题游戏体力变现模型，不确定在小程序能跑通
3. 小程序的流量来源不能买量

**AI 回应**：

- 第1点对：小游戏激励视频 eCPM 碾压小程序，原因是用户心理预期不同
- 第2点：小程序激励视频触发场景被严格审查，小游戏同样的文案是平台鼓励的标准做法；IAP体力瓶在小游戏审核几乎无阻力
- 第3点：小程序确实几乎不能买量，小游戏可以全量微信广告投放

**建议先在小程序侧验证变现，2-4周上架跑数据，DAU和ARPU出来后再决定是否值得迁小游戏。用户未回复是否有历史数据。

## 五、变现模式讨论

**用户确认**：小游戏的变现有两种：看广告和充钱。两者要求的资质不一样。

**AI 回应**：

- 纯广告：只需要公司+微信认证+软著，2个月能搞定
- IAP内购：还需要版号（6-18个月，¥15-30万代办）+文网文
- 答题类产品可以走“教育”类目规避版号，但微信审核越来越严
- 建议路线：Phase 1 纯广告上线 → Phase 2 同时申请版号 → Phase 3 版号到手开 IAP

**用户强调**：纯广告游戏，不需要分析功能。自己有数。

## 六、小游戏跳转公众号问题

**用户问**：小游戏可以跳转到公众号吗？

**AI 回答**：

- 可以，但是间接的
- 最常用：客服消息 → 自动回复公众号关注卡片，转化5-15%
- 小游戏不支持 `<official-account>` 组件（没有 WXML）
- 为什么限制：微信不鼓励把高价值广告流量导出到低抽成场景
- 反向是开放的：公众号 → 小游戏

## 七、产品模式确认

**用户总结产品模式**：

1. 迁移到小游戏，原因：高广告收入，前期只靠广告
2. 模式：答题道具赛季制、答题消耗体力、有排位、正确得100保底+时间得分+书籍加成，错误0分。道具按科目来，书籍增加得分。答题概率掉抽奖券，抽奖得书籍。每个赛季1个月，换题库、清排位、清书籍
3. 变现靠看广告得体力，体力有自然恢复和广告恢复两种

**AI 确认理解并反馈**：

- 核心循环：体力驱动答题 → 答题产出分数+抽奖券+书籍 → 书籍反哺答题 → 排位验证 → 赛季重置
- 变现点：单点纯广告，体力是唯一瓶颈
- 确认边界：是否没有金币体系？体力参数是否沿用旧的？书籍是攻防属性还是纯加分？
- 观察：新设计比旧项目精简很多，但缺少日常打卡理由，抽奖券掉落率是关键杠杆

**用户回复**：金币抽奖券无所谓。金币系统是后期做皮肤变现的，皮肤可以用金币买，抽奖券也可以用金币买。这个后面再说。

## 八、商业模式与定位深化

**用户提出核心观点**：

1. 这是一个重运营的项目，广告只能维持项目本身
2. 架构捕获垂类群体，比如金融知识问答、K12知识问答
3. 两个走向：自运营（垂类矩阵）或作为营销工具卖给别人

**自运营模式**：多个垂类小游戏 → 将用户收集到群 → 社群运营变现

**卖工具模式**：套代码+维护费，这对技术架构有多租户要求

**AI 分析**：

- 两个方向本质上通向同一个技术架构：一套引擎、多套配置、多端分发
- 做的不再是“一个小游戏”，而是“一个答题小游戏生成器”
- 两个方向不互斥，先自运营跑通一个垂类，再复制或产品化

**用户明确**：方向1，一套代码、多套题库、多个用户数据库。但上架是多个小游戏，即多个软著。自运营优先，除非判断自运营吃不下了，再作为营销工具。

**用户关注点**：私域引流是必然的，垂类选择优先看变现路径清晰度，前期相当于提供线索就行

**商业定位一句话总结**：“用答题做垂类人群筛选器。前端是小游戏，后端是线索交易。”

## 十、重构规划与数据库摸底（2026-06-22）

**用户需求**：准备重构项目，先了解旧系统。

### 数据库环境搭建

- 旧备份 `wxdkgamedo_20190618_data.sql`（MySQL 5.7，96 张表）导入本地
- MSI 安装器卡依赖 → 改用 ZIP 手动部署 `mysqld --initialize-insecure`
- 导入必须设 `max_allowed_packet=1G`，否则大 INSERT 报 `gone away`
- 11 张表备份缺失，从 Java DAO 反推结构补建
- 最终 107 张表全部就绪，生成了三份数据库文档：说明、数据统计、表关联关系

### API 接口摸底

- 扫描全部 51 个 Controller，234 个接口
- 前台 57 个（`/vientianetest/mobile/`），后台 177 个
- 旧 API 设计问题：全 POST、手动拼 JSON、一个接口干两件事、无类型安全
- 提炼出「业务能力清单」——剥离旧 URL/参数，纯业务语言描述 18 大模块、50+ 项业务能力

### 技术选型讨论

**用户决策**：前端后端全部重写，全栈 JS/TS。

- 前端选 Phaser.js 而非 Cocos Creator，理由：学习成本低（3-5 天 vs 2-3 周）、AI 配合效率高、答题游戏不需要 Cocos 的编辑器复杂度
- 后端 Node.js（框架待定：Express/NestJS/Fastify），ORM 待定
- 数据库 MySQL 照用，表结构做精简调整
- 微信小游戏适配：确认 Phaser + adapter.js 可行，适配器本质是补缺失浏览器 API 的 polyfill，非黑魔法
- 前后端通过 shared/types.ts 共享接口定义，类型安全

### 功能裁剪对比（旧版 vs V4 商业计划）

**砍掉的**（约 10 项）：撸猫、每周挑战、专项练习、门票、留言板、微信模板消息、跨小程序、金币系统延迟、8 种排行榜精简为段位榜、双套抽奖精简为答题掉券一条链路

**补上的**（约 5 项）：广告系统（激励视频+插屏）、线索变现漏斗、多项目配置化、冷启动获客、软著/合规

**修正的**（约 8 项）：得分公式（100+速度分+书籍加成+末题×30）、体力参数（30上限/1局1点/120s恢复/广告+5）、赛季清零范围、抽奖链路、学科加成、段位数量（9段位青铜→王者）、机器人用途简化、排行榜种类缩减

### MVP 排期估算（配合 AI）

## 十一、管理后台评估与排期修正（2026-06-22）

### 讨论起因

用户问：现在 AI 时代了还需要后台吗？旧后台 renren-fast 的 177 个接口（18 个 Controller）本质全是 CRUD + 分页搜索 + 图片上传——为不会写 SQL 的运营人员准备的 Web 表单。

### 结论：管理后台可以砍掉

- renren-fast 177 个接口 → 完全不用做
- 前端不需要对应的管理页面
- 出题/配置/查数据 → DBeaver + AI 生成脚本直接操作数据库
- 旧架构"人→Web表单→Controller→Service→DAO→SQL"变成"你描述需求→AI生成脚本→SQL"

### 砍后台后的修订排期

**后端（Node.js + TS）**：5-7 周（省 CRUD 约 1 周）

| 模块 | 周数 |
|------|------|
| 项目脚手架 + DB Schema + Redis | 1 周 |
| 用户认证（mock + 微信） | 0.5 周 |
| 题库查询引擎 | 1 周 |
| 对战引擎（WebSocket） | 2-3 周 |
| 排位 + 赛季 | 1 周 |
| 体力系统 | 0.5 周 |
| 学科书籍 + 抽奖 | 0.5 周 |
| 段位排行榜 | 0.5 周 |
| 广告回调 | 0.5 周 |

**前端（Phaser.js）**：8-12 周（不受后台影响）

### 三次估算对比

| | 完整版（全功能） | 砍功能（V4裁剪） | 砍后台（本次） |
|------|---------|----------|----------|
| 后端 | 8 周 | 6-8 周 | **5-7 周** |
| 前端 | 12 周 | 8-12 周 | **8-12 周** |
| 合计 | 20 周 | 14-20 周 | **13-19 周** |
| 配合 AI | 5 个月 | 4 个月 | **3-3.5 个月** |

每天 4 小时 × 配合 AI → **约 3 个月出头**能出大众版 MVP。

### 砍后台的核心收益

不是代码量（省了约 1 周），而是：
1. 不用学/维护 renren-fast 那套框架
2. 以后出题运营直接用 DBeaver + AI 脚本，不用等前端页面
3. 新增功能不需要再写 Controller→Service→DAO 的三层 CRUD
